# Отчёт об исправлении ошибок и оптимизации проекта broken-app

## 1. Поиск и исправление багов

### 1.1. Выход за границу среза в `sum_even`

Цикл `0..=values.len()` итерировался на один элемент дальше конца среза. В сочетании с `get_unchecked` это приводило к Undefined Behavior при каждом вызове. Miri зафиксировал нарушение: `Undefined Behavior: 'assume' called with 'false'`.

**Исправление:** unsafe-блок удалён, функция переписана на безопасный итератор `values.iter().copied().filter(|v| v % 2 == 0).sum()`.

### 1.2. Use-after-free в `use_after_free`

Указатель читался после передачи в `Box::from_raw` и освобождения памяти — гарантированное UB.

**Исправление:** функция удалена как не имеющая безопасного применения.

### 1.3. Утечка памяти в `leak_buffer`

Память, выделенная через `Box::into_raw`, никогда не освобождалась обратным вызовом `Box::from_raw`. Valgrind показал `definitely lost: 5 bytes`.

**Исправление:** unsafe-блок удалён, функция заменена на `input.iter().filter(|b| **b != 0).count()`.

### 1.4. Неверная логика `average_positive`

Усреднение выполнялось по всем элементам среза без фильтрации. Тест `averages_only_positive` падал: для `[-5, 5, 15]` выдавал 5 вместо ожидаемых 10.

**Исправление:** добавлен фильтр `filter(|v| *v > 0)`, после которого сумма и количество считаются только по положительным значениям.

### 1.5. Неполная обработка пробельных символов в `normalize`

Функция удаляла только символ пробела `' '`, игнорируя табуляции, переносы строк и другие пробельные символы.

**Исправление:** `replace(' ', "")` заменён на `split_whitespace().collect::<String>()`.

### 1.6. Гонка данных в `race_increment`

Глобальная переменная `static mut COUNTER` изменялась из нескольких потоков без атомарных операций или блокировок. Классическая гонка данных — UB.

**Исправление:** `static mut u64` заменён на `static AtomicU64`, инкремент через `fetch_add(1, Ordering::SeqCst)`.

## 2. Оптимизация

### 2.1. `slow_fib` — переход от O(2ⁿ) к O(n)

Исходная рекурсивная реализация без мемоизации порождает экспоненциальное число вызовов. На `n=35` выполняется порядка 30 миллионов рекурсивных вызовов. Профилировщик показал, что функция занимает 73.7% всего времени выполнения.

**Сделанные изменения:** рекурсия заменена итеративным проходом с двумя накопительными переменными. Сложность снижена до O(n), накладные расходы на вызовы исчезли.

**Результат:** ускорение примерно в 347 000 раз на `n=32` (с 5.83 мс до 16.8 нс по criterion). Функция перестала отображаться в профилировщике.

### 2.2. `slow_dedup` — переход от O(n²) к O(n log n)

Для каждого элемента выполнялся линейный поиск по уже добавленным уникальным значениям, после каждой вставки вызывалась сортировка. На 10 тысячах элементов время составляло около 13 мс.

**Сделанные изменения:** уникальность отслеживается через `HashSet` со вставкой за O(1), сортировка выполняется один раз в конце. Итоговая сложность — O(n log n).

**Результат:** ускорение примерно в 90 раз на 10K элементах (с 9.70 мс до 104.5 мкс по criterion).

## 3. Регрессионные тесты

Добавлено 12 тестов, покрывающих каждый исправленный баг:

- `sum_even`: пустой срез, все нечётные, отрицательные и граничные значения
- `leak_buffer`: пустой вход, все нули
- `normalize`: табуляции и переносы строк, множественные пробелы внутри текста
- `average_positive`: пустой срез, все отрицательные, ровно один элемент
- `race_increment`: 4 потока по 1000 итераций — ровно 4000
